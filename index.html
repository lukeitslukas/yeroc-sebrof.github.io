From my Third year at Abertay I worked as part of a group to complete a generic development project. Working as part of a Five person team, we were given a set of briefs and client requirements that we had to either develop a tool for or perform an assessment. Our team had already put a great deal of throught into out project before the Term had started; we planned on developing a physical penetration device framework for the Raspberry Pi Zero W. From a Hardware perspective, our project let us play about with a Pi USB stem and a Blinkt LED GPIO kit which is something we've never played much with. We called our end product Skeleton Key (SK).

I really enjoyed the mix of hardware and software development; this project was a perfect excuse for all of us to learn Python 3.6 and low level Linux internals. Our SK project, which can be found on GitHub, makes use of the Debian OS kernel modules to emulate generic HID devices when connected to a system via the data USB port on the Pi or through the USB Stem. This framework was developed to simplify this communication with these OS modules and allows for Python 3 scripts to be hotswapped to perform a range of actions. We also chose Python because we wanted an accessable framework for these modules that anyone could pick up and have a go with themselves.

For this framework we made four modules as examples in a hope to encourage further development. The first module we developed performed network enumeration, this made use of common tools for penetration testers to scan a predetermined range of devices and generate a easy to read report using HTML and Bootstrap.

Responder, the second module, integrated the SpiderLabs tool Responder to attempt to exfiltrate password hashes from enterprise Windows PC's. SK will then store these internally for later use from the user as the Pi does not have adequate password cracking performance.

The next module, Storage, allows the user to make use of existing file systems or generate a new file systems to mount over the USB bus. This module was mostly developed to display the frameworks ability to perfom this action and is to be used as part of a larger script.

Finally, KeyInject was developed to show the frameworks ability to emulate a Keyboard and the inbuild DuckyScript parser. The KeyInject module tied together these two sections of SKs functionality, the framework will read a DuckyScript file and execute the contained instructions.

Overall, this was a really enjoyable project that let me experience working alongside an excelent team. I am very proud of not only my work but that of all members of the team. It's a shame that the weeks following this projects completion the Kernel modules regarding Keystroke injection were completely overhauled within Debian so this project could do with a bit of fixing.

I decided with the year of research ahead of me that I wanted to development a tool that makes use of General Purpose Graphical Processing Unit (GPGPU) assets. To meet this goals I focused my investigation of the file carving process into accelerating the file carving header/footer search using GPGPUs. This turned out to be a much larger challenge than I originally intended as many of these projects do. Development was difficult and I was unable to reach my original goal of a full GPU enabled file carver.
Regardless of the difficulty endured the final product, Aletheia, was able to well outperform existing CPU bound Open Source File carvers when they were in audit mode (A mode where the actual carving out of the files is not performed). Furthermore, as the only other existing tool for GPGPU File Carving is written to only operate on the Windows operating system I was pleased that this project is able to be developed into the first cross platform file carver for GPGPU technology. I intend to continue development in the years to come as this project means alot to me with the hope that it could be used in a law enforcement capacity.
