Skeleton Key - Group Project

The biggest module(s) from my 3rd year was my group project, this a team of coursemates to coordinate a professional development project. For this, my team of four and I, decided to make a physical penetration device framework for the Raspberry Pi Zero W. We also made use of a USB stem and a Blinkt LED kit to make our device more accessable. We called our end product Skeleton Key (SK).

I really enjoyed the mix of phyiscal and software development and this was the perfect excuse for me to learn Python 3.6. This project, which can be found on GitHub, makes use of the Debian OS' kernel modules. This allows us to emulate generic devices when connected to a device via USB. Our framework lets us communicate with these OS modules, using the mentioned Python 3.6. We wanted an accessable framework so we make use of a module system so users can use Python code to develop payloads.

For this framework we made four modules as examples; The first module being enumeration, made use of common tools for penetration testers to scan a predetermined range of devices and generate a easy to read report using HTML and Bootstrap. Responder, the second module, makes use of SpiderLabs Responder to take password hashes from enterprise windows PC's and stores them on SK for later use. The next module, Storage, can make use of or generate a new file system which will be mounted over the USB connection to the device SK connected to. Finally, KeyInject makes use of our frameworks ability to both emulate a Keyboard and parse DuckyScript to read a user selected DuckyScript file and execute the contained instructions.

Overall, this was a really enjoyable project that let me experience working alongside an excelent team. I am very proud of not only my work but that of all members of the team. It's a shame that the weeks following this projects completion the Kernel modules were completely overhauled within Debian so this project could do with a bit of redevelopment.

Aletheia File Header/Footer search - Dissertation

I decided with the year of research ahead of me that I wanted to development a tool that makes use of General Purpose Graphical Processing Unit (GPGPU) assets. To meet this goals I focused my investigation of the file carving process into accelerating the file carving header/footer search using GPGPUs. This turned out to be a much larger challenge than I originally intended as many of these projects do. Development was difficult and I was unable to reach my original goal of a full GPU enabled file carver.
Regardless of the difficulty endured the final product, Aletheia, was able to well outperform existing CPU bound Open Source File carvers when they were in audit mode (A mode where the actual carving out of the files is not performed). Furthermore, as the only other existing tool for GPGPU File Carving is written to only operate on the Windows operating system I was pleased that this project is able to be developed into the first cross platform file carver for GPGPU technology. I intend to continue development in the years to come as this project means alot to me with the hope that it could be used in a law enforcement capacity.
